INFORME DE RESULTADOS: PRUEBAS DE INTEGRACIÓN API - DEMOBLAZE

RESUMEN EJECUTIVO

Se realizó el análisis y ejecución de una suite de pruebas automatizadas para los endpoints de autenticación (/signup y /login) de la API de Demoblaze. El objetivo fue evaluar la integridad funcional, el manejo de excepciones y la continuidad del flujo de datos (End-to-End). La ejecución arrojó una tasa de éxito del 100% (8/8 escenarios superados) con un tiempo total de validación de ~11 segundos.

HALLAZGOS PRINCIPALES
Métricas de Ejecución y Cobertura

Performance Base: Los tiempos de respuesta (Response Time) se mantuvieron consistentes, permitiendo validar 8 flujos de negocio complejos en menos de 12 segundos usando el AllTestsRunner.

Idempotencia Asegurada: Se implementó inyección dinámica de datos. Esto garantizó que cada ejecución usara emails/usuarios únicos, demostrando que la suite no requiere limpieza de datos previa (Data Setup/Teardown) para ejecutarse en pipelines CI/CD.

Análisis del Contrato de la API 

Anomalía Arquitectónica en Signup: Al intentar crear un usuario ya existente, la API maneja la excepción correctamente enviando el mensaje "This user already exists.". Sin embargo, el servidor responde con un Status Code 200 OK en lugar del código estándar HTTP 409 Conflict o 400 Bad Request.

Anomalía en Login: Similar al registro, los errores de autenticación ("Wrong password", "User does not exist") retornan Status 200 OK en lugar de 401 Unauthorized.

Integración End-to-End

Continuidad de Datos: El escenario 3.4 demostró persistencia inmediata. El sistema fue capaz de crear un usuario dinámico en /signup e inmediatamente utilizar ese mismo token de datos en /login, validando la integración real entre ambos servicio o controladores.

CONCLUSIONES
Estabilidad Funcional: El sistema de registro y login de Demoblaze es estable y maneja las excepciones de usuario sin presentar caídas del servidor (HTTP 5xx), confirmando que las validaciones internas funcionan.

Deuda Técnica en Estándares REST: El uso generalizado del HTTP 200 para flujos de error representa una mala práctica arquitectónica. Esto obliga a los clientes (Frontend/Mobile) a parsear el cuerpo del JSON para saber si hubo un error, en lugar de confiar en las cabeceras HTTP, dificultando el monitoreo con herramientas genéricas (APMs).

Madurez de la Suite de Automatización: La arquitectura basada en Karate DSL construida para este ejercicio es modular y altamente escalable, permitiendo la integración continua mediante runners unificados.

RECOMENDACIONES
Refactorización de Status Codes: Requerir al equipo de desarrollo Backend alinear las respuestas a los estándares REST (400, 401, 409) para mejorar la observabilidad del sistema y la facilidad de consumo del Front-end.

Implementación de Políticas de Seguridad: Dado el bajo tiempo de respuesta de los endpoints y la ausencia de bloqueos observados durante las pruebas, se sugiere implementar validaciones como Rate Limiting (Limitación de tasa) o CAPTCHAs para mitigar posibles ataques de fuerza bruta en el Login.
